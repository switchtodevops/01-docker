The LABEL instruction in Dockerfile adds metadata to Docker images as key-value pairs. Labels are essential for documentation, automation, and organization.

Basic Syntax

LABEL <key>=<value> <key>=<value> ...

Common Use Cases

Basic Labels

LABEL version="1.0.0" LABEL maintainer="devops@company.com" LABEL description="Production web server"

Basic metadata about the image

OCI Standard Labels

LABEL org.opencontainers.image.created="2024-02-15" \ org.opencontainers.image.version="1.0.0" \ org.opencontainers.image.authors="DevOps Team" \ org.opencontainers.image.description="Web Application"

Industry-standard OCI labels for better compatibility

Multi-line Labels

LABEL version="1.0.0" \ build-date="2024-02-15" \ maintainer="devops@company.com" \ description="Production API Server"

Organize multiple labels for better readability

View Labels

# View all labels docker inspect myimage:tag | jq '.[0].Config.Labels' # View specific label docker inspect myimage:tag | jq -r '.[0].Config.Labels["version"]' # List all labels as key=value docker inspect myimage:tag | jq -r '.[0].Config.Labels | to_entries[] | "\(.key)=\(.value)"'

Filter by Labels

# Filter images by label docker images --filter "label=version=1.0.0" # Filter containers by label docker ps --filter "label=environment=production"

Complete Example

FROM nginx:alpine LABEL org.opencontainers.image.created="2024-02-15T10:00:00Z" \ org.opencontainers.image.version="1.0.0" \ org.opencontainers.image.authors="DevOps Navigator" \ org.opencontainers.image.title="Web Server" \ org.opencontainers.image.description="Production web server" \ org.opencontainers.image.vendor="Company Inc" \ org.opencontainers.image.license="MIT" COPY index.html /usr/share/nginx/html/ EXPOSE 80 CMD ["nginx", "-g", "daemon off;"]

Common Label Categories

  • Version Information: version, build-date, git-commit
  • Maintainer Info: maintainer, team, contact
  • Documentation: description, documentation-url, source-url
  • Environment: environment, deployment-stage
  • Compliance: license, vendor, security-scan-status
  • Build Info: build-tool, ci-build-number
Pro Tip: Use OCI standard labels (org.opencontainers.image.*) for better compatibility with tools and registries.

Dynamic Labels with Build Args

ARG BUILD_DATE ARG VERSION ARG GIT_COMMIT FROM nginx:alpine LABEL build.date="${BUILD_DATE}" \ version="${VERSION}" \ git.commit="${GIT_COMMIT}" # Build command: # docker build \ # --build-arg BUILD_DATE=$(date -u +"%Y-%m-%dT%H:%M:%SZ") \ # --build-arg VERSION=1.0.0 \ # --build-arg GIT_COMMIT=$(git rev-parse HEAD) \ # -t myapp:1.0.0 .
Best Practice: Use labels for metadata, monitoring, and automation. Never store sensitive information like passwords in labels.

Benefits of Labels

  • Documentation: Self-documenting images
  • Automation: CI/CD can read labels for decisions
  • Organization: Filter and group images
  • Monitoring: Track versions in production
  • Compliance: License and security tracking

The docker tag command creates a new tag for an existing image. Tags are used to identify and version Docker images.

Basic Syntax

docker tag SOURCE_IMAGE[:TAG] TARGET_IMAGE[:TAG]

Common Commands

Tag with version

docker tag myapp:latest myapp:1.0.0

Creates a new tag "1.0.0" pointing to the same image as "latest"

Tag for Docker Hub

docker tag myapp:1.0.0 username/myapp:1.0.0

Prepares image for pushing to Docker Hub with username

Tag for private registry

docker tag myapp:1.0.0 registry.company.com/myapp:1.0.0

Tags image for pushing to private registry

Multiple tags

docker tag myapp:1.0.0 myapp:1.0 docker tag myapp:1.0.0 myapp:1 docker tag myapp:1.0.0 myapp:latest

Creates multiple tags for different versioning levels

Tagging Best Practices

  • Semantic Versioning: Use MAJOR.MINOR.PATCH format (1.0.0)
  • Git Commit SHA: Tag with commit hash for traceability
  • Environment Tags: Use dev, staging, prod for environments
  • Date Tags: Include build date (2024-02-15)
  • Avoid "latest" in Production: Always use specific versions
Pro Tip: Use multiple tags to support different versioning strategies. For example, tag as both "1.2.3" and "1.2" and "1" for flexibility.

Tagging Strategy Example

# Build the image docker build -t myapp:latest . # Tag with semantic version docker tag myapp:latest myapp:1.2.3 # Tag with minor version docker tag myapp:latest myapp:1.2 # Tag with major version docker tag myapp:latest myapp:1 # Tag with git commit docker tag myapp:latest myapp:abc123 # Tag for Docker Hub docker tag myapp:latest username/myapp:1.2.3
Best Practice: In CI/CD pipelines, automatically tag images with build number, git commit SHA, and semantic version for complete traceability.
